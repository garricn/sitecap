# CLI-1: --dry-run flag

## Context

sitecap currently requires a full browser launch + page navigation + capture pipeline to discover what a site contains. For planning and inventory purposes, users need a lightweight mode that crawls, discovers URLs, detects CMS metadata, and counts media/resource references — without writing screenshots, HTML, accessibility trees, or downloading any files.

## Goal

Add `--dry-run` flag that runs the crawl/discovery pipeline but skips all file-writing capture steps. Outputs a JSON inventory to stdout (or file via `-o`).

## Design

### Behavior

`--dry-run` implies `--crawl` (if not already set) and requires a browser connection (any mode — CDP attach, `--profile`, or `--launch`). If no browser is available and `--launch` is not set, error with the same message as normal mode. It navigates pages, extracts links, optionally detects CMS — but never calls `capturePage()`.

| Flag combo | Behavior |
|---|---|
| `--dry-run` | Crawl from seed URL(s), output inventory JSON |
| `--dry-run --max-pages 5` | Crawl limited set |
| `--dry-run -t cms` | Include CMS detection per page |
| `--dry-run --no-crawl` | Single page inventory (no link following) — NOT supported initially, `--dry-run` always crawls |

### Inventory JSON schema

```json
{
  "seed": "https://example.com",
  "timestamp": "2026-03-18T...",
  "duration_ms": 1234,
  "pages": [
    {
      "url": "https://example.com/",
      "slug": "example.com",
      "depth": 0,
      "links": ["https://example.com/about", "..."],
      "resources": { "stylesheet": 3, "script": 5, "image": 12, "font": 2 },
      "cms": { "detected": true, "cms": "wordpress", "version": "6.4", "confidence": "high", "indicators": [...], "admin": false }
    }
  ],
  "summary": {
    "totalPages": 15,
    "totalResources": { "stylesheet": 8, "script": 12, "image": 45, "font": 3 },
    "cms": "wordpress"
  }
}
```

- `cms` field only present when `-t cms` is included. Pass through the full `detectCms()` result — `confidence`, `indicators`, `context`, and `warnings` are all useful for inventory purposes.
- `resources` counted from network requests (by `resourceType`) — requires `setupNetworkCapture()` but NOT `capturePage()`
- `links` are same-origin extracted links (from `extractLinks()`)

### Output

- Default: JSON to stdout (pipe-friendly)
- With `-o <dir>`: writes `inventory.json` to the output dir

## Files to modify

### `bin/sitecap.js`
- Add `"dry-run": { type: "boolean", default: false }` to parseArgs options
- Add `--dry-run` to help text
- After browser/context setup and before the worker loop: if `--dry-run`, run a separate dry-run loop instead of the capture worker loop
- Dry-run loop per page (order matters — mirrors `navigateAndCapture()` minus the capture):
  1. `setupNetworkCapture(page, { networkFilter: 'all' })` — **must be before navigation** so response listener is attached
  2. `page.goto(url, { waitUntil: 'domcontentloaded', timeout: 30_000 })`
  3. `waitForPageSettle(page)`
  4. `extractLinks(page)` → store in page entry
  5. Count resources from `page.__sitecapNetwork` grouped by `resourceType`
  6. If `-t cms`: `detectCms(page, page.__sitecapNetwork)` → store full result (including `confidence`, `indicators`, `context`, `warnings`)
  7. Never call `capturePage()`
- **stdout must stay clean for JSON**: all progress/status lines (`console.log`) go to `process.stderr` in dry-run mode. This keeps stdout pipe-friendly for `sitecap --dry-run | jq`.
- After loop: build summary, output JSON to stdout or write to `outDir/inventory.json`
- Skip all session-video, asset download, and manifest code paths when dry-run
- Do NOT call `setupConsoleCapture()` — console data is not part of the inventory schema and would be wasted work

### `lib/capture.js`
- No changes needed — `setupNetworkCapture()`, `setupConsoleCapture()`, `waitForPageSettle()`, and `extractLinks()` are already exported and usable independently of `capturePage()`

### `lib/cms.js`
- No changes — `detectCms()` already works with a page + network array

## Key decisions

1. **dry-run loop lives in `bin/sitecap.js`**, not in a new module. It's a CLI-only concern — the library API doesn't need a dry-run mode (callers can just call `extractLinks` directly).
2. **Resource counting** uses `page.__sitecapNetwork` entries grouped by `resourceType`. This means we still navigate + settle each page, but skip all file I/O.
3. **CMS detection is opt-in** via `-t cms` — consistent with existing behavior where CMS is opt-in.
4. **No `--download-assets`, `--download-media`, `--video`, `--session-video`** in dry-run mode — these are silently ignored (or error if you prefer; silent ignore is simpler).

## Inversion-prone logic

The core branch is: `if (dryRun) { runDryRunLoop() } else { runCaptureWorkers() }`. No inversion risk — it's a clean early branch before the worker loop.

| dryRun | crawl | behavior |
|--------|-------|----------|
| true   | any   | Always crawls (dry-run implies crawl) |
| false  | true  | Normal crawl + capture |
| false  | false | Normal single/multi-page capture |

## Validation

### Automated (local, `make check`)

- **CLI test**: `--dry-run` with local test server → assert stdout is valid JSON matching inventory schema
- **CLI test**: `--dry-run -t cms` → assert `cms` field present in page entries
- **CLI test**: `--dry-run` → assert NO files written to output dir (no screenshot.png, no meta.json)
- **CLI test**: `--dry-run --max-pages 3` → assert `pages.length <= 3`
- **CLI test**: `--dry-run` help text includes `--dry-run`

### Test assertions

```
assert JSON.parse(stdout).pages.length > 0
assert JSON.parse(stdout).summary.totalPages === JSON.parse(stdout).pages.length
assert !existsSync(join(outDir, '127.0.0.1', 'screenshot.png'))
assert !existsSync(join(outDir, '127.0.0.1', 'meta.json'))
assert JSON.parse(stdout).pages[0].resources !== undefined
assert JSON.parse(stdout).pages[0].links instanceof Array
```
